# 第二阶段学习指南：和第一阶段有何不同

> 🎯 写给刚完成 Python 基础的你

---

## 一、学习方式的根本转变

### 第一阶段（Python 基础）→ 第二阶段（Web 开发）

| 对比维度 | 第一阶段 | 第二阶段 |
|----------|----------|----------|
| **学习对象** | 语言本身（语法、数据结构） | 协议 + 框架 + 工具（HTTP、FastAPI、SQL） |
| **练习方式** | 写函数、做算法题 | 搭服务、发请求、建数据库 |
| **调试方式** | print / 断点 | **浏览器 F12 / API 测试工具 / 日志** |
| **运行方式** | `python xxx.py` 跑完就结束 | 启动服务器 → 持续运行 → 用客户端访问 |
| **错误来源** | 语法错误、逻辑错误 | 网络问题、端口占用、协议格式、跨域... |
| **知识特点** | 确定性高（对就是对） | 约定性高（RESTful 是规范，不是语法） |
| **成果形态** | 命令行程序 | **可访问的 Web API / 网站** |

### 🧠 核心心态转变

```
第一阶段：「我学会了这个语法，能做对题目」
    ↓
第二阶段：「我理解了这个协议/框架，能搭一个可访问的服务」
```

## 二、第二阶段特有的学习技巧

### 1. 🌐 先理解"为什么"再学"怎么用"

第一阶段学 `for` 循环，直接写就行。
但第二阶段学 HTTP 方法，你需要先理解：
- **为什么**要分 GET 和 POST？（安全性和幂等性）
- **为什么**要设计状态码？（标准化错误处理）
- **为什么**要有请求头？（元数据传递）

> 💡 每个技术背后都有一个"解决什么问题"的故事。先找到这个故事。

### 2. 🔧 工具链变多了

第一阶段你只需要：
- Python + IDE + 终端

第二阶段你需要熟悉：
| 工具 | 用途 | 什么时候用 |
|------|------|-----------|
| **浏览器 F12** | 查看网络请求 | 学 HTTP、调试前后端 |
| **Postman / Apifox** | 发送 API 请求测试 | 测试自己写的 API |
| **curl** | 命令行发 HTTP 请求 | 快速测试、脚本中 |
| **Swagger UI** | FastAPI 自动生成的 API 文档 | 调试和展示你的 API |
| **数据库可视化工具** | 查看数据库内容 | 学 SQL、调试数据问题 |
| **Git** | 版本控制 | 项目级代码管理 |

> 💡 不用一次全学，遇到什么学什么。但 **F12 和 Postman 是第一天就要用的**。

### 3. 🔄 从"写脚本"到"跑服务"

```python
# 第一阶段：脚本模式
def add(a, b):
    return a + b

print(add(1, 2))  # 跑完就退出了
```

```python
# 第二阶段：服务模式
from fastapi import FastAPI
app = FastAPI()

@app.get("/add")
def add(a: int, b: int):
    return {"result": a + b}

# 启动后持续运行，等待别人来访问
# uvicorn main:app --reload
```

**关键区别**：你的代码变成了一个"服务"，其他人（或前端）通过网络来调用它。

### 4. 📖 官方文档成为最好的老师

- 第一阶段：看教程、看视频就够了
- 第二阶段：**官方文档是第一手资料**
  - [FastAPI 中文文档](https://fastapi.tiangolo.com/zh/) ← 质量极高
  - [requests 文档](https://requests.readthedocs.io/zh-cn/latest/)
  - [SQLAlchemy 文档](https://docs.sqlalchemy.org/)

> 💡 视频教程帮你入门，但遇到问题一定要查官方文档。

### 5. 🐛 Debug 方式完全不同

| 问题类型 | 第一阶段 | 第二阶段 |
|----------|----------|----------|
| 程序报错 | 看 Traceback，定位行号 | 看服务器日志 + 客户端错误信息 |
| 结果不对 | print 变量 | **F12 看请求/响应内容** |
| 连不上 | 不存在这个问题 | 检查端口、防火墙、URL、CORS |
| 数据不对 | 检查算法逻辑 | 检查数据库、请求体格式 |

## 三、Day 22 学习建议（HTTP 协议基础）

### 推荐学习流程（6 小时分配）

```
第 1 小时：📖 通读学习笔记
    - 重点理解 URL 结构、HTTP 方法、状态码
    - 不求记住所有请求头，但要知道它们的存在

第 2 小时：🌐 动手观察真实 HTTP
    - 打开 Chrome F12 → Network 标签
    - 访问 baidu.com、github.com 等网站
    - 观察每个请求的方法、状态码、请求头
    - 特别关注：登录请求、搜索请求

第 3 小时：💻 运行练习题
    - pip install requests
    - 运行 homework22.py，逐个完成练习
    - 观察每个请求的发送和响应

第 4 小时：🛠️ 用 httpbin.org 实验
    - 手动在浏览器访问 https://httpbin.org/get?name=test
    - 用 Postman/Apifox 发送 POST 请求
    - 尝试设置不同的请求头

第 5 小时：📝 整理笔记
    - 用自己的话总结 GET vs POST
    - 画一个 HTTP 请求-响应的完整流程图
    - 记录常见状态码和含义

第 6 小时：🧠 思考题 + 拓展
    - 完成 homework22.py 的练习 9
    - 选读：Cookie 和 Session 的工作原理
    - 选读：HTTPS 的加密原理（了解即可）
```

### 关键概念检查

学完 Day 22 后，你应该能回答这些问题：

1. **在浏览器地址栏输入 `https://www.google.com/search?q=python` 后发生了什么？**
   （DNS 解析 → 建立 TCP 连接 → TLS 握手 → 发送 HTTP GET 请求 → 服务器处理 → 返回 HTML → 浏览器渲染）

2. **GET 和 POST 的核心区别是什么？**
   （不只是"参数位置不同"，要理解幂等性、安全性、缓存策略的差异）

3. **为什么 401 和 403 是两个不同的状态码？**
   （"我不知道你是谁" vs "我知道你是谁但不让你进"）

4. **HTTP 无状态是什么意思？实际开发中怎么解决？**
   （Cookie+Session 或 Token）

## 四、第二阶段常见踩坑提醒

### 1. 端口被占用
```
OSError: [Errno 98] Address already in use
```
**解决**：`netstat -ano | findstr 8000` 找到进程，kill 掉，或换个端口

### 2. CORS 跨域问题
```
Access to fetch at 'http://localhost:8000' from origin 'http://localhost:3000' 
has been blocked by CORS policy
```
**解决**：FastAPI 中配置 `CORSMiddleware`（Day 34 会详细学）

### 3. JSON 格式问题
```
422 Unprocessable Entity
```
**解决**：检查请求体格式是否和 Pydantic Model 定义一致

### 4. 忘记启动服务器
```
Connection refused
```
**解决**：确保 `uvicorn main:app --reload` 在运行

### 5. requests 超时
```
TimeoutError / ConnectionError
```
**解决**：检查网络，国内可能需要代理访问某些 API

## 五、LeetCode 同步推进

第二阶段要求 LeetCode 累计 50 题（新增 30 道 Easy）。

建议每天学完 Web 内容后，花 30-45 分钟做 1-2 道 LeetCode。

**推荐顺序**：
- Week 4：栈/队列相关（用 Python 的 list 和 collections.deque）
- Week 5：双指针（快慢指针、左右指针）
- Week 6：滑动窗口、链表操作

> 刷 LeetCode 不仅是面试需要，也能保持 Python 语法的熟练度。

## 六、一句话总结

> 第一阶段是**练内功**（Python 语法），第二阶段是**学招式**（Web 框架）。
> 内功要扎实，但招式要在**实战**中学——多搭服务、多发请求、多看报错信息。
>
> 从今天开始，你写的每个程序，都可以变成一个"服务"，让别人通过网络来访问。
> **这就是 Web 开发的魅力。** 🚀
