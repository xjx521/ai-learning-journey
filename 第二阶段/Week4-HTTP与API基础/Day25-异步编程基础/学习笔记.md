## 一、为什么需要异步？

### 🍔 生活化类比

**同步（Synchronous）= 排队点餐**
```
你点餐 → 等待厨房做好 → 你取餐 → 下一个人点餐
问题：你在等待时什么都不能做
```

**异步（Asynchronous）= 拿号等叫号**
```
你点餐拿到号 → 去旁边喝饮料/玩手机 → 叫到你的号 → 你去取餐
好处：等待的时间里你可以做别的事
```

### 在 Web 开发中

```
同步服务器（如 Flask 默认）：
  请求 A 来了 → 查数据库（等 100ms）→ 返回 → 处理请求 B
  问题：等数据库时，请求 B 只能等着

异步服务器（如 FastAPI）：
  请求 A 来了 → 开始查数据库 → [等待期间去处理请求 B] → 数据库返回了 → 继续处理 A
  好处：等待 I/O 时可以服务其他请求
```

**关键**：异步的优势在于 **I/O 密集型** 任务（查数据库、调 API、读写文件），不在于 CPU 密集型（计算、排序）。

## 二、同步 vs 异步代码对比

### 同步代码（按顺序执行）

```python
import time

def buy_coffee(name):
    print(f"[{name}] 点咖啡...")
    time.sleep(2)  # 模拟等待咖啡制作（阻塞！）
    print(f"[{name}] 咖啡好了！")
    return f"{name}的咖啡"

def buy_bread(name):
    print(f"[{name}] 买面包...")
    time.sleep(1)  # 模拟等待（阻塞！）
    print(f"[{name}] 面包好了！")
    return f"{name}的面包"

# 同步执行
start = time.time()
buy_coffee("张三")   # 等 2 秒
buy_bread("张三")    # 再等 1 秒
print(f"总共花了 {time.time() - start:.1f} 秒")
# 输出：总共花了 3.0 秒（2 + 1）
```

### 异步代码（并发执行）

```python
import asyncio

async def buy_coffee_async(name):
    print(f"[{name}] 点咖啡...")
    await asyncio.sleep(2)  # 模拟等待（非阻塞！期间可以做别的事）
    print(f"[{name}] 咖啡好了！")
    return f"{name}的咖啡"

async def buy_bread_async(name):
    print(f"[{name}] 买面包...")
    await asyncio.sleep(1)  # 非阻塞等待
    print(f"[{name}] 面包好了！")
    return f"{name}的面包"

# 异步并发执行
async def main():
    start = time.time()
    # 同时开始两个任务
    coffee_task = buy_coffee_async("张三")
    bread_task = buy_bread_async("张三")
    await asyncio.gather(coffee_task, bread_task)
    print(f"总共花了 {time.time() - start:.1f} 秒")
    # 输出：总共花了 2.0 秒（max(2, 1)）

asyncio.run(main())
```

**核心区别**：
- `time.sleep(2)` → 整个程序卡住 2 秒
- `await asyncio.sleep(2)` → 只暂停当前协程，其他协程可以运行

## 三、核心概念

### 1. 协程（Coroutine）

```python
# 普通函数
def hello():
    return "hello"

# 协程函数（加了 async）
async def hello_async():
    return "hello"

# 调用普通函数 → 直接得到结果
result = hello()  # "hello"

# 调用协程函数 → 得到一个协程对象（不会执行！）
coro = hello_async()  # <coroutine object>

# 要真正执行协程，需要 await 或 asyncio.run()
result = asyncio.run(hello_async())  # "hello"
```

**类比**：协程就像一个可以"暂停"和"恢复"的函数。

### 2. async 和 await

| 关键字 | 作用 | 用在哪 |
|--------|------|--------|
| `async def` | 定义协程函数 | 函数定义时 |
| `await` | 等待异步操作完成，期间让出控制权 | 协程函数内部 |

```python
async def fetch_data():
    # await 只能在 async def 内部使用！
    data = await some_async_api_call()  # 暂停这里，让出控制权
    return data
```

**规则**：
- `await` 只能在 `async def` 内部使用
- `await` 后面跟的是一个"可等待对象"（协程、Task、Future）

### 3. 事件循环（Event Loop）

```
事件循环就像一个调度员：

1. 协程 A 开始执行
2. A 遇到 await → 暂停 → 调度员记录"A 在等 XX"
3. 调度员让协程 B 开始执行
4. B 遇到 await → 暂停
5. A 等待的操作完成了 → 调度员恢复 A 的执行
6. ... 循环往复
```

`asyncio.run(main())` 会创建并启动一个事件循环。

### 4. asyncio.gather() — 并发执行多个协程

```python
async def task_a():
    await asyncio.sleep(1)
    return "A 完成"

async def task_b():
    await asyncio.sleep(2)
    return "B 完成"

async def main():
    # 同时开始，等所有任务完成
    results = await asyncio.gather(task_a(), task_b())
    print(results)  # ["A 完成", "B 完成"]
    # 总耗时 = max(1, 2) = 2 秒
```

## 四、常见误区

### ❌ 误区 1：async 让代码变快

**真相**：async 不会让单次操作变快，它只是在等待时去做别的事。

```
同步：A 等 2 秒 → B 等 1 秒 = 总共 3 秒
异步：A 等 2 秒（期间 B 也开始等 1 秒）= 总共 2 秒

但如果只有一个任务：
同步：A 等 2 秒 = 2 秒
异步：A 等 2 秒 = 还是 2 秒（没有别的任务可以切换）
```

### ❌ 误区 2：async 适合 CPU 密集任务

**真相**：如果任务一直在算（没有 I/O 等待），async 没有优势。

```python
# 这个用 async 没有意义（纯计算，没有等待）
async def compute():
    return sum(i * i for i in range(1000000))

# 这个用 async 有意义（有 I/O 等待）
async def fetch():
    return await http_client.get("https://api.example.com/data")
```

### ❌ 误区 3：在异步中调用同步阻塞代码

```python
# ❌ 错误：在 async 函数里调用阻塞的 time.sleep
async def bad():
    time.sleep(2)  # 这会阻塞整个事件循环！所有协程都卡住

# ✅ 正确：用 asyncio.sleep
async def good():
    await asyncio.sleep(2)

# ✅ 如果必须调用阻塞代码，用 run_in_executor
import concurrent.futures
async def run_blocking():
    loop = asyncio.get_event_loop()
    result = await loop.run_in_executor(None, time.sleep, 2)
```

## 五、异步和 FastAPI 的关系

### FastAPI 同时支持同步和异步

```python
from fastapi import FastAPI
app = FastAPI()

# 同步路由（适合 CPU 密集或简单逻辑）
@app.get("/sync")
def sync_endpoint():
    return {"message": "同步处理"}

# 异步路由（适合 I/O 密集：查数据库、调 API）
@app.get("/async")
async def async_endpoint():
    data = await some_async_db_query()
    return {"message": "异步处理", "data": data}
```

### 什么时候用 async def？

| 场景 | 用 `def` | 用 `async def` |
|------|----------|----------------|
| 纯计算、不涉及 I/O | ✅ | ❌ 无意义 |
| 查数据库（同步 ORM） | ✅ | ❌ 会阻塞事件循环 |
| 查数据库（异步 ORM） | ❌ | ✅ |
| 调用外部 API | ❌ | ✅ |
| 读写文件 | ✅（或 async + aiofiles） | ✅（配合 aiofiles） |
| 不确定 | ✅ 安全选择 | 需要确认内部没有阻塞调用 |

**新手建议**：刚开始用 FastAPI 时，如果不确定，就用普通 `def`。等你开始用异步数据库驱动或异步 HTTP 客户端时，再切换到 `async def`。
