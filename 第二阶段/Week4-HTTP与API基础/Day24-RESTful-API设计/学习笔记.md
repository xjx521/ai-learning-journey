## 一、什么是 RESTful API？

**REST**（Representational State Transfer，表述性状态转移）是一种 **API 设计风格**，不是技术标准。

### 🍔 类比理解

```
非 RESTful（随意命名）：
  POST /createUser        ← 创建用户
  POST /deleteUser        ← 也是 POST？
  GET  /getUserInfo       ← 获取用户
  POST /updateUserAge     ← 更新年龄

RESTful（规范命名）：
  POST   /users           ← 创建用户（方法表示动作）
  DELETE /users/123       ← 删除用户（URL 表示资源）
  GET    /users/123       ← 获取用户
  PATCH  /users/123       ← 更新用户部分信息
```

**核心区别**：
- 非 RESTful：URL 里带动词（`/createUser`），方法随意
- RESTful：**URL 只表示资源（名词），用 HTTP 方法表示动作**

### RESTful 的 URL = 名词，HTTP 方法 = 动词

```
GET    /users       → "给我用户列表"     （读）
POST   /users       → "创建新用户"       （增）
GET    /users/123   → "给我 123 号用户"  （读单个）
PUT    /users/123   → "替换 123 号用户"  （全量改）
PATCH  /users/123   → "修改 123 号用户的某个字段"（部分改）
DELETE /users/123   → "删除 123 号用户"  （删）
```

## 二、RESTful API 设计规则

### 规则 1：URL 用名词，不用动词

| ❌ 不推荐 | ✅ 推荐 | 说明 |
|-----------|--------|------|
| `GET /getUsers` | `GET /users` | URL 不需要动词 |
| `POST /createUser` | `POST /users` | POST 本身就是"创建" |
| `POST /deleteUser?id=1` | `DELETE /users/1` | 用 DELETE 方法 + 路径参数 |
| `GET /searchUsers?q=xx` | `GET /users?q=xx` | 搜索用查询参数 |

### 规则 2：名词用复数形式

| ❌ 不推荐 | ✅ 推荐 |
|-----------|--------|
| `/user/123` | `/users/123` |
| `/article` | `/articles` |
| `/product/5/comments` | `/products/5/comments` |

### 规则 3：嵌套表示关系

```
获取用户 123 的所有文章：
  GET /users/123/articles

获取用户 123 的第 5 篇文章：
  GET /users/123/articles/5

给用户 123 的第 5 篇文章添加评论：
  POST /users/123/articles/5/comments
```

⚠️ 嵌套不要超过 2 层，否则 URL 太长：
```
❌ /users/123/articles/5/comments/78/replies
✅ /comments/78/replies （直接通过评论 ID 访问）
```

### 规则 4：用查询参数做过滤、排序、分页

```
过滤：
  GET /users?role=admin              → 只查管理员
  GET /articles?status=published     → 只查已发布

排序：
  GET /users?sort=created_at&order=desc   → 按创建时间倒序

分页：
  GET /users?page=1&size=20          → 第 1 页，每页 20 条
  或
  GET /users?offset=0&limit=20       → 偏移量 0，限制 20 条

组合使用：
  GET /articles?status=published&sort=views&page=1&size=10
```

### 规则 5：版本号放在 URL 或请求头中

```
URL 版本（推荐）：
  GET /api/v1/users
  GET /api/v2/users

请求头版本：
  GET /users
  Accept: application/vnd.myapi.v2+json
```

### 规则 6：返回统一的数据格式

```json
// ✅ 成功响应
{
    "code": 200,
    "message": "success",
    "data": {
        "id": 1,
        "name": "张三"
    }
}

// ✅ 失败响应
{
    "code": 400,
    "message": "用户名不能为空",
    "data": null
}

// ✅ 列表响应（带分页信息）
{
    "code": 200,
    "message": "success",
    "data": [
        {"id": 1, "name": "张三"},
        {"id": 2, "name": "李四"}
    ],
    "pagination": {
        "page": 1,
        "size": 20,
        "total": 100,
        "total_pages": 5
    }
}
```

## 三、HTTP 状态码的正确使用

RESTful 风格要求**用对状态码**，不要全部返回 200：

| 场景 | 状态码 | 响应体 |
|------|--------|--------|
| 获取成功 | 200 | 资源数据 |
| 创建成功 | 201 | 新建的资源（含 ID） |
| 删除成功 | 204 | 无内容（或不返回 body） |
| 参数校验失败 | 422 | 具体哪个字段错了 |
| 资源不存在 | 404 | 错误信息 |
| 未认证 | 401 | 提示登录 |
| 无权限 | 403 | 提示权限不足 |
| 服务器错误 | 500 | 通用错误信息（不暴露细节） |

### ⚠️ 常见错误：全部返回 200

```python
# ❌ 不推荐
@app.get("/users/999")
def get_user(user_id: int):
    user = db.get(user_id)
    if not user:
        return {"code": 404, "message": "用户不存在"}  # 但 HTTP 状态码是 200！

# ✅ 推荐
from fastapi import HTTPException

@app.get("/users/{user_id}")
def get_user(user_id: int):
    user = db.get(user_id)
    if not user:
        raise HTTPException(status_code=404, detail="用户不存在")
    return user  # HTTP 状态码也是 404
```

## 四、JSON 数据格式

### JSON 语法复习

```json
{
    "name": "张三",          // 字符串必须用双引号
    "age": 21,               // 数字不需要引号
    "is_student": true,      // 布尔值：true / false（小写）
    "courses": ["Python", "HTTP"],  // 数组
    "address": {             // 嵌套对象
        "city": "北京",
        "zip": "100000"
    },
    "girlfriend": null       // 空值：null（小写）
}
```

### JSON vs Python 字典

| 对比 | JSON | Python dict |
|------|------|-------------|
| 布尔值 | `true` / `false` | `True` / `False` |
| 空值 | `null` | `None` |
| 键名 | 必须双引号 | 可以单引号 |
| 本质 | 字符串（文本格式） | 内存中的数据结构 |

```python
import json

# Python dict → JSON 字符串（序列化）
data = {"name": "张三", "is_student": True, "gf": None}
json_str = json.dumps(data, ensure_ascii=False)
# '{"name": "张三", "is_student": true, "gf": null}'

# JSON 字符串 → Python dict（反序列化）
parsed = json.loads(json_str)
# {"name": "张三", "is_student": True, "gf": None}
```

## 五、API 文档规范（Swagger / OpenAPI）

### 什么是 OpenAPI？

**OpenAPI Specification（OAS）** 是一种描述 API 的标准格式（YAML 或 JSON）。

```yaml
# 一个 OpenAPI 描述示例
paths:
  /users:
    get:
      summary: 获取用户列表
      parameters:
        - name: page
          in: query
          type: integer
      responses:
        200:
          description: 成功返回用户列表
```

有了这个描述文件，就可以自动生成：
- 📖 可读的 API 文档网页
- 🧪 测试工具
- 💻 各语言的 SDK 代码

### FastAPI 的杀手级特性：自动文档

FastAPI **自动生成** OpenAPI 文档，不需要手写：

```python
from fastapi import FastAPI

app = FastAPI(title="我的 API", version="1.0.0")

@app.get("/users/{user_id}", summary="获取用户信息")
def get_user(user_id: int):
    """通过用户 ID 获取详细信息"""
    return {"user_id": user_id}
```

启动后访问：
- **Swagger UI**：`http://localhost:8000/docs` （交互式测试）
- **ReDoc**：`http://localhost:8000/redoc` （漂亮文档页）
- **OpenAPI JSON**：`http://localhost:8000/openapi.json` （机器可读）

**这意味着**：你写代码的同时，文档就自动生成了。不用再单独写 API 文档！
