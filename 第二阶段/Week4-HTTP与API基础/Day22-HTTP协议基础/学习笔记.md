## 一、HTTP 是什么？

**HTTP（HyperText Transfer Protocol，超文本传输协议）** 是互联网上最基础的通信协议。

### 🍔 生活化类比

把 HTTP 想象成**去餐厅点餐**：
- **你（客户端/浏览器）** → 看菜单、点菜
- **服务员（HTTP 协议）** → 把你的请求告诉厨房
- **厨房（服务器）** → 做好菜
- **服务员（HTTP 协议）** → 把菜端给你

HTTP 就是那个**规定"怎么点菜、怎么上菜"的服务员**。

## 二、URL 结构（网址的解剖）

```
https://www.example.com:8080/api/users?page=1&limit=10#section1
│         │               │    │         │            │
│         │               │    │         │            └─ 片段标识符（锚点）
│         │               │    │         └─ 查询参数（Query String）
│         │               │    └─ 路径（Path）
│         │               └─ 端口号（Port）
│         └─ 域名（Domain / Host）
└─ 协议（Scheme / Protocol）
```

### 各部分详解

| 部分 | 示例 | 说明 |
|------|------|------|
| **协议** | `https://` | `http`（明文）或 `https`（加密，多一层 SSL/TLS） |
| **域名** | `www.example.com` | 服务器的地址，DNS 会把它翻译成 IP 地址 |
| **端口** | `:8080` | 服务器上具体哪个"窗口"在监听。HTTP 默认 80，HTTPS 默认 443 |
| **路径** | `/api/users` | 要访问的具体资源，像文件系统的目录 |
| **查询参数** | `?page=1&limit=10` | 附加条件，用 `&` 连接多个键值对 |
| **片段** | `#section1` | 页面内定位，**不会发给服务器** |

### 💡 小贴士
- `?` 是查询参数的开始标记，只有一个
- `&` 用来连接多个参数
- 查询参数里的空格会被编码成 `%20` 或 `+`

## 三、HTTP 请求方法（动作）

HTTP 定义了一套标准的"动作"，告诉服务器你想做什么：

### 核心四大方法（CRUD 对应）

| 方法 | 全称 | 做什么 | CRUD | 生活类比 | 是否有请求体 |
|------|------|--------|------|----------|-------------|
| **GET** | Get | **获取**资源 | Read | "我要看菜单" | ❌ 无 |
| **POST** | Post | **创建**资源 | Create | "我要点一个新菜" | ✅ 有 |
| **PUT** | Put | **全量更新**资源 | Update | "把 3 号桌的订单全部换成新的" | ✅ 有 |
| **DELETE** | Delete | **删除**资源 | Delete | "退掉 3 号菜" | ❌ 通常无 |

### 其他常见方法

| 方法 | 做什么 | 使用场景 |
|------|--------|----------|
| **PATCH** | **部分更新**资源 | 只改用户头像，其他信息不变 |
| **OPTIONS** | 询问服务器支持哪些方法 | 浏览器跨域请求前的"预检" |
| **HEAD** | 和 GET 一样，但**只返回响应头** | 检查文件是否存在、大小、修改时间 |

### ⚠️ GET vs POST 的重要区别

```
GET：
- 参数放在 URL 里（?key=value）
- 有长度限制（浏览器通常 2048 字符）
- 会被浏览器缓存、收藏、分享
- **绝对不要用来传密码等敏感信息**
- 幂等（Idempotent）：请求 1 次和 100 次，服务器状态不变

POST：
- 参数放在请求体（Body）里
- 理论上无长度限制
- 不会被缓存
- 非幂等：每次请求可能创建新资源
```

### 🔑 幂等性（Idempotency）

> "幂等"意思是：**不管你执行多少次，结果都一样**。

- GET、PUT、DELETE 是幂等的
- POST **不是**幂等的（重复提交可能创建多条记录）
- 这就是为什么**刷新页面时 POST 请求会弹出警告**

## 四、HTTP 请求结构

一个完整的 HTTP 请求长这样：

```
POST /api/users HTTP/1.1                    ← 请求行：方法 + 路径 + 协议版本
Host: www.example.com                       ← 请求头：目标域名
Content-Type: application/json              ← 请求头：数据格式
Authorization: Bearer abc123token           ← 请求头：认证信息
User-Agent: Mozilla/5.0 (Windows NT 10.0)   ← 请求头：客户端信息
                                            ← 空行（分隔头和体）
{                                           ← 请求体（Body）
    "name": "张三",
    "email": "zhangsan@example.com"
}
```

### 常见请求头（Request Headers）

| 请求头 | 作用 | 示例值 |
|--------|------|--------|
| **Host** | 目标域名（HTTP/1.1 必填） | `www.example.com` |
| **Content-Type** | 请求体的数据格式 | `application/json`、`text/html`、`multipart/form-data` |
| **Accept** | 客户端能接受的响应格式 | `application/json`、`text/html` |
| **Authorization** | 认证凭证 | `Bearer <token>`、`Basic <base64>` |
| **User-Agent** | 客户端标识 | `Mozilla/5.0 ...` |
| **Cookie** | 客户端存储的会话信息 | `session_id=abc123` |
| **Cache-Control** | 缓存策略 | `no-cache`、`max-age=3600` |
| **Origin** | 请求来源（跨域时重要） | `http://localhost:3000` |

## 五、HTTP 响应结构

服务器的回复长这样：

```
HTTP/1.1 200 OK                             ← 状态行：协议 + 状态码 + 原因短语
Content-Type: application/json              ← 响应头
Content-Length: 128                         ← 响应头
Set-Cookie: session=xyz; Path=/             ← 响应头
                                            ← 空行
{                                           ← 响应体
    "id": 1,
    "name": "张三",
    "email": "zhangsan@example.com"
}
```

### 常见响应头（Response Headers）

| 响应头 | 作用 | 示例值 |
|--------|------|--------|
| **Content-Type** | 响应体的数据格式 | `application/json` |
| **Content-Length** | 响应体大小（字节） | `128` |
| **Set-Cookie** | 让客户端存 Cookie | `session=xyz; Path=/; HttpOnly` |
| **Location** | 重定向目标 URL（3xx 时使用） | `/login` |
| **Cache-Control** | 缓存指令 | `public, max-age=86400` |
| **Access-Control-Allow-Origin** | 允许跨域的源 | `*` 或 `http://localhost:3000` |

## 六、HTTP 状态码（服务器的"表情包"）

状态码是 3 位数字，第一位表示大类：

### 📗 2xx 成功（"好的！"）

| 状态码 | 含义 | 什么时候用 |
|--------|------|-----------|
| **200** | OK | 请求成功（GET 最常用） |
| **201** | Created | 创建成功（POST 成功后） |
| **204** | No Content | 成功但没内容返回（DELETE 常用） |

### 📙 3xx 重定向（"去别的地方看看"）

| 状态码 | 含义 | 什么时候用 |
|--------|------|-----------|
| **301** | Moved Permanently | 永久迁移（旧网址换新网址） |
| **302** | Found | 临时重定向 |
| **304** | Not Modified | 没修改过，用缓存就行 |

### 📕 4xx 客户端错误（"你的问题"）

| 状态码 | 含义 | 什么时候用 |
|--------|------|-----------|
| **400** | Bad Request | 请求格式不对（参数缺失、JSON 格式错误） |
| **401** | Unauthorized | 没登录 / Token 过期 |
| **403** | Forbidden | 登录了但没权限（普通用户想访问管理员页面） |
| **404** | Not Found | 资源不存在（URL 打错了） |
| **405** | Method Not Allowed | 方法不对（该用 POST 的用了 GET） |
| **409** | Conflict | 资源冲突（注册时用户名已存在） |
| **422** | Unprocessable Entity | 格式对但数据不合法（邮箱格式错、年龄为负数） |
| **429** | Too Many Requests | 请求太频繁（限流） |

### 📓 5xx 服务器错误（"我的问题"）

| 状态码 | 含义 | 什么时候用 |
|--------|------|-----------|
| **500** | Internal Server Error | 服务器内部出错（代码有 Bug） |
| **502** | Bad Gateway | 网关/代理出问题 |
| **503** | Service Unavailable | 服务器忙/维护中 |
| **504** | Gateway Timeout | 上游服务器响应超时 |

### 🧠 记忆技巧

```
2xx = "成功了 ✅"
3xx = "去别处 ↪️"
4xx = "你错了 ❌"（客户端问题）
5xx = "我错了 💥"（服务器问题）

200 = OK
201 = 创建了
400 = 你请求写错了
401 = 你是谁？（没认证）
403 = 我知道你是谁，但不行（没权限）
404 = 找不到
500 = 服务器炸了
```

## 七、HTTP 的无状态性与 Cookie/Session

### 无状态（Stateless）

HTTP 是**无状态的**——每次请求都是独立的，服务器不会"记住"你上次做了什么。

**类比**：每次去柜台办业务，柜员都不认识你，需要你出示身份证。

### 那怎么实现"登录状态"？

```
方案一：Cookie + Session
1. 用户登录 → 服务器创建 Session，返回 Set-Cookie: session_id=abc
2. 浏览器自动存 Cookie
3. 以后每次请求自动带上 Cookie: session_id=abc
4. 服务器通过 session_id 找到用户信息

方案二：Token（JWT）
1. 用户登录 → 服务器生成 Token 返回
2. 客户端存在 localStorage
3. 以后每次请求手动带上 Authorization: Bearer <token>
4. 服务器验证 Token 签名
```

在后续 FastAPI 开发中，我们主要用 **JWT Token** 方案。

## 八、HTTPS 与 HTTP 的区别

| 对比项 | HTTP | HTTPS |
|--------|------|-------|
| 安全性 | 明文传输，可被窃听 | SSL/TLS 加密，安全 |
| 端口 | 80 | 443 |
| URL 前缀 | `http://` | `https://` |
| SEO | 浏览器会标"不安全" | 搜索引擎优先收录 |
| 速度 | 略快（无加密开销） | 略慢（但 HTTP/2 弥补了） |

**结论**：生产环境**必须用 HTTPS**。开发阶段可以用 HTTP。

## 九、实战：用浏览器看 HTTP 请求

### 步骤
1. 打开 Chrome，按 `F12` 打开开发者工具
2. 切到 **Network（网络）** 标签
3. 访问任意网站（如 `https://httpbin.org`）
4. 点击列表中的任意请求
5. 查看：
   - **Headers** 标签 → 请求头、响应头、状态码
   - **Response** 标签 → 响应体
   - **Timing** 标签 → 各阶段耗时

### 推荐测试网站
- `https://httpbin.org` — 专门测试 HTTP 请求的网站，会返回你发送的内容
- `https://jsonplaceholder.typicode.com` — 假数据 API，支持 GET/POST/PUT/DELETE
